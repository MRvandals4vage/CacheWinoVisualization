01 — The Algorithm

What is Winograd convolution?

Standard convolution is arithmetic-heavy. Winograd's insight: factor the computation to reuse intermediate products, slashing multiply-accumulate operations by up to 2.25×.

Direct convolution — sliding a 3×3 kernel over a 5×5 input
For every output pixel, the kernel slides across the input performing 9 multiplications + 8 additions = 17 MACs. For a 3×3 kernel producing a 3×3 output tile: 9 × 9 = 81 multiply-accumulate operations.
Direct MACs = Cin × Cout × H' × W' × K²
For K=3: each output element costs 9 MACs

02 — The Problem

So why doesn't it work
in practice?

The arithmetic wins are real. But edge CPUs are not compute-bound — they're memory-bound. And Winograd quietly makes the memory problem worse.

📦

Data inflation

Every input tile expands from m² to (m+2)² elements. For F(2,3), that's a 4× intermediate blowup in working-set size.

🗃️

Cache pressure

Inflated intermediate tensors spill out of L1/L2 onto DRAM. On a Jetson Nano with 2 MB L2, this happens constantly.

⚡

Bandwidth cost

DRAM reads are 100× slower than L1 cache. Every cache miss erases the arithmetic gain Winograd promised.

Memory amplification αmem(m) = (m+2)² / m² — intermediate data blowup per tile
The F(2,3) scheme inflates each tile's data by 4×. Larger tile sizes (m=6) improve arithmetic efficiency but still inflate by 1.78×. On devices with 1–2 MB L2 caches, this routinely causes working sets to exceed cache capacity entirely.
The real culprit: static scheduling

✗ No explicit model of working-set vs cache capacity

✗ No per-layer adaptation — one size fits all

03 — The Solution

CacheWinograd:
three interlocked fixes

The framework treats Winograd tile selection as a cache-constrained scheduling problem — solved at runtime, per layer, in nanoseconds.

Cache Probing
Read L1/L2 sizes from sysfs at init. Zero offline work.
PHASE 02
Tile Selection
Per layer: solve constrained optimisation in O(1) time.
PHASE 03
Fused Execution
Channel-outer fusion eliminates intermediate tensor writes.

Phase 2 — Adaptive tile selection (per layer, ~5 ns)
For each candidate tile size, compute extended working-set including NEON spill buffers:
WSext(m) = β(m+2)²Cin(1+g) + βm²Cout + 2β(m+2)²
where β=4 (FP32), g=4 (128-bit NEON)
Select the tile that maximises reuse efficiency subject to fitting in cache:
m* = argmax Cout·m² / (m+2)² subject to WSext(m) ≤ γ·Ccache
